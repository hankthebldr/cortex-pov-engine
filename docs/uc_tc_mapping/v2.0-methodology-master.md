# Cortex Use-Case Index v2.0 — Master Detection Methodology

**Source:** `HR - Cortex Use Case Index - Master v2.0.xlsx`
**Coverage:** 248 Test Cases (123 SecOps + 125 Cloud), 56 Use Cases, 79 MOAT-tier, 79 LEAD-tier, 90 PARITY-tier
**Status:** Active framework — drives CortexSim scenario design and POV deliverables
**Owner:** Henry Reed · Last updated 2026-05-27

---

## 1 · Purpose

This document defines **how every test case in the v2.0 index becomes an achievable, instrumented POV deliverable in CortexSim**. It is the bridge between the sales-side TC index (what the customer wants validated) and the engineering-side scenario library (what SimCore actually executes).

The 248 TCs do not require 248 bespoke detection methodologies. When clustered by their `Primary KPI` and `Detection Source` columns, they collapse into **10 methodology families**, each with one reusable validation harness in SimCore. ~60 % of all TCs are served by two detection sources alone (`XSIAM UI + XQL`, `Cortex Cloud Posture Dashboard + REST API`), so most TCs only need a scenario YAML + an `expected_detections` block — no custom code.

---

## 2 · The 10 methodology families

Every TC is assigned to exactly one family. The family determines the validation harness, the KPI capture path, and the success-criteria template.

| ID | Family | KPI signature | TC count | Primary harness |
|----|--------|---------------|---------:|-----------------|
| **F1** | Signal Injection & Detection Accuracy | MTTD, Detection Accuracy, False Positive Rate, URL detection rate | **82** | EAL plugin → known-signature event → XQL count + timing |
| **F2** | Causality Stitching Validation | Cross-Source Correlation Rate, Causality Chain Completeness | **27** | Multi-plane orchestrator → single incident assertion via `xdr_data` causality view |
| **F3** | Asset Discovery & Inventory Coverage | Asset Discovery Coverage, Discovery Latency, SaaS coverage % | **23** | Plant N assets → poll `/api/v1/inventory` → discovered / planted ratio |
| **F4** | Posture & Compliance Scoring | Compliance Score, Coverage %, Control dimension coverage, Pipeline gate accuracy | **34** | Plant N misconfigs → fetch posture score → finding-count parity |
| **F5** | Automation & Workflow | Automation Rate, Triage Automation Rate, Remediation Success Rate, Volume reduction % | **43** | Trigger event → wait for playbook completion → state diff |
| **F6** | Response Timing | MTTR, Response time, SLA Compliance, Behavioral / drift latency | **19** | Time-anchor execute → poll for response signal → delta in seconds |
| **F7** | AI Triage & Summarization | AI Summary Accuracy, Investigation quality score | **1** | Labelled alert corpus → AI classification → confusion matrix |
| **F8** | Integration & Ingestion Health | Completeness %, Deployment time, Policy deployment time | **16** | Onboard source → schema-fidelity check → field-presence count |
| **F9** | Code-to-Cloud Traceability | Traceability Chain Completeness | **1** | Tag commit → image → pod → assert chain across all 3 |
| **F10** | Qualitative Evidence | Outcome Achieved (DC-attested) | **2** | Operator-attested checkbox with screenshot capture |

**Why this clustering matters:** F1 + F2 + F3 + F4 = 166 TCs (67 %) reuse one of four well-defined harness shapes. The remaining 82 TCs split across automation, timing, ingestion — all of which CortexSim's existing orchestrator already supports via the `expected_detections` + `mttd_seconds` mechanism in `core/models.py`.

The full per-TC mapping (TC ID → family → plane → scenario file → KPI → threshold) is in [`v2.0-tc-mapping-table.csv`](v2.0-tc-mapping-table.csv) — 248 rows, machine-queryable, the canonical source of truth for which TCs map to which scenarios.

## 3 · Family-by-family methodology recipes

For each family below: the **validation pattern** (what SimCore does), the **measurement method** (how the KPI is captured), and the **scenario YAML shape** (what fields are required beyond the existing schema).

### F1 — Signal Injection & Detection Accuracy

**Pattern.** SimCore (or an agent) emits a controlled signal with a known TTP signature into the customer environment. After a bounded wait, an XQL probe counts how many matching incidents/alerts appeared. KPI = (matched / injected) and `incident.creation_time - signal.emit_time`.

```
emit(signal_set, signature) → wait(bounded) → xql_count(filter=signature) → assert(matched ≥ threshold ∧ mttd ≤ threshold)
```

**Measurement method.** Every step's `expected_detections` row becomes a `Result` ORM record on Run creation (`core/engine/orchestrator.py` already does this). DC marks observed via `PUT /api/results/{id}/validate`, which sets `observed_at` and computes `mttd_seconds` automatically. For automated KPI capture, the validation harness should run a saved XQL query on a timer and call the same `validate` endpoint when the count threshold trips.

**Scenario shape addendum.** Each step needs at least one `expected_detections` row with `verification_xql` populated (new field — see §5). Example:

```yaml
steps:
  - id: step-01
    command: |
      curl -A 'CortexSimBeacon/1.0' http://testmynids.org/uid/index.html
    expected_detections:
      - plane: NDR
        type: Analytics
        description: "NGFW — outbound to known IOC list"
        verification_xql: |
          dataset = pan_ngfw_traffic
          | filter dst_host = "testmynids.org" and bytes_sent > 0
          | fields source_ip, dst_host, _time
        threshold: { kpi: MTTD, op: "≤", value: 60, unit: seconds }
```

**Best fit TCs.** TC-IR-01, TC-NDR-04, TC-EDR-01..06, TC-DLP-*, TC-EMAIL-*, TC-AIRS-01..05, TC-CRES-* runtime alerts. Anything where the question is "did Cortex fire on the thing we injected, fast enough."

### F2 — Causality Stitching Validation

**Pattern.** Multiple coordinated signals across ≥2 planes, sharing a session/host/identity key. KPI is the rate at which Cortex stitches them into a single incident with a complete causality chain — that's the moat against best-of-breed point products.

```
emit(NDR_signal) ∥ emit(EDR_signal) ∥ emit(ITDR_signal)  [same src_host, ±60s window]
→ poll(/incidents)
→ assert(single_incident_id ∧ causality_chain.length ≥ N ∧ all_planes_present)
```

**Measurement method.** Verification XQL targets the Causality View (`dataset=xdr_data`), counting distinct planes represented under one `incident_id`. The KPI `Cross-Source Correlation Rate` = % of test runs where stitching succeeded within the time budget.

**Scenario shape.** This is the existing `plane: ANALYTICS` pattern in `scenarios/multi_plane/*.yml`. The schema already supports it; the deep-dive doc [`methodology-multi-plane-stitching.md`](methodology-multi-plane-stitching.md) catalogs every F2 TC against a SIM-MP-* scenario.

**Best fit TCs.** TC-IR-05, TC-NDR-01/02, TC-ERV-02, TC-CITH-07, TC-ITDR-* MITM/lateral chains. Full F2 inventory in the deep-dive.

### F3 — Asset Discovery & Inventory Coverage

**Pattern.** Stand up a known-cardinality set of discoverable assets (EC2 instances, SageMaker endpoints, Bedrock models, public-facing services, AD accounts, SaaS OAuth grants). Wait for Cortex Cloud / XPANSE / AI-SPM inventory scan. Compare discovered count to planted count.

```
provision(N assets with tag=CortexSim) → wait(discovery_cycle) → GET /api/v1/inventory?tag=CortexSim → coverage = found/N
```

**Measurement method.** Coverage % and Discovery Latency. The IaC modules (`infra/modules/aws/{cspm,asm,ai-spm}/`) plant tagged assets; the validation harness polls the Cortex Cloud REST API. SimCore needs a new `inventory_probe` validation method that runs the poll on a timer — see §5.

**Best fit TCs.** TC-AISP-01, TC-CASM-01, TC-ASM-01..04, TC-CIEM-* coverage TCs, TC-PSER-01 multi-cloud onboarding, TC-AGRP-01..03 application grouping. Full F3 inventory in [`methodology-ai-spm.md`](methodology-ai-spm.md) for the AI-SPM subset.

### F4 — Posture & Compliance Scoring

**Pattern.** Plant a known set of misconfigurations with deterministic policy mapping. Wait for posture scan. Compare detected-finding count to planted-finding count, broken out by severity and control framework.

```
deploy(misconfig_pack with policy_ids) → wait(scan_interval) → GET /api/v1/findings?source=CortexSim → coverage = matched/planted
```

**Measurement method.** Today the `infra/modules/aws/cspm/` module plants 9 well-known findings (public S3, SG with SSH world-open, weak CloudTrail, etc.) each tagged `CortexSimCSPMFinding=<type>`. The methodology requires extending the same pattern to KSPM, DSPM, CIEM, and IaC scan modules so each TC has a deterministic finding-count baseline.

**Best fit TCs.** TC-CSPM-*, TC-KSPM-*, TC-DSPM-*, TC-CIEM-*, TC-IACS-01..04, TC-SSCAN-01..05 (secrets scanning), TC-COMP-*, TC-ASPM-* policy enforcement.

### F5 — Automation & Workflow

**Pattern.** Trigger an event that should kick off an XSOAR or XSIAM playbook. Wait for terminal state. Compare resulting system state to the playbook's declared intent.

```
fire(trigger_event) → poll(incident.status until in [closed, contained, escalated]) → assert(actions_taken match playbook spec)
```

**Measurement method.** Automation Rate = (auto-handled incidents / total fired). Triage Automation Rate = (auto-classified-correctly / total). The harness needs to inspect the incident's audit trail via XQL on `dataset=incidents` and count playbook-driven actions vs. analyst-driven actions.

**Best fit TCs.** TC-IR-01..03 (auto-triage, grouping, scoring), TC-SOAR-01..03, TC-CRWF-* (Cloud Remediation Workflow), TC-CMTR-* (MTTR family), TC-ITPA-01.

### F6 — Response Timing

**Pattern.** Anchor a precise emit timestamp. Poll for a downstream artifact (containment action, MFA challenge, NGFW EDL update, ServiceNow ticket). Compute delta. Apply SLA threshold.

```
t0 = emit(trigger) ; while not poll(response): sleep ; t1 = now ; assert(t1 - t0 ≤ SLA)
```

**Measurement method.** Same `mttd_seconds` field on the Result table, repurposed as `mttr_seconds` or `response_latency_seconds`. SimCore's existing timestamp infrastructure handles this — no schema change.

**Best fit TCs.** TC-CMTR-* (Coordinated Response MTTR), TC-CMTD-* (Visibility & AI Driven Detection MTTD), TC-AISP-* drift detection, TC-EDR-06 (containment), TC-MDR-01.

### F7 — AI Triage & Summarization

**Pattern.** Feed a labeled corpus of incidents (mix of known malicious + benign noise) into the tenant. Compare AI's classifier output / summary against the ground-truth labels.

```
ingest(labeled_corpus) → wait(triage_cycle) → fetch(incident.ai_score, incident.ai_summary) → compute(precision, recall, F1) vs. labels
```

**Measurement method.** Requires a labeled-corpus fixture (currently planned for `scenarios/multi_plane/packages/` but not yet built). The single F7 TC (TC-CITH-15 AI Summary Accuracy) is a stretch goal — the AI quality measurement methodology should reuse the AIRS scorer pattern from `sources/cortex-prompt-attacker/` (Probe → Mutator → Scorer pipeline).

**Best fit TCs.** TC-CITH-15. Limited but high-visibility because it directly attacks Charlotte AI's marketing claims.

### F8 — Integration & Ingestion Health

**Pattern.** Onboard a new log source / IdP / Marketplace pack. Validate schema fidelity: do the expected fields appear, are they typed correctly, does volume match the source's reported throughput.

```
onboard(source) → wait(ingest_cycle) → xql(dataset=<source> | fields *) → check(field_presence, type_inference, volume vs. source counters)
```

**Measurement method.** Completeness % = % of expected schema fields present and typed correctly. Deployment time = `onboarded_at - initiated_at`.

**Best fit TCs.** TC-IR-08 (Broker VM 3+ sources), TC-NDR-05 (Broker VM 3rd-party network), TC-PSER-01 (multi-cloud onboarding), TC-DEV-* (developer experience integrations), TC-CGOV-* governance integrations.

### F9 — Code-to-Cloud Traceability

**Pattern.** Tag a git commit with a marker. Build → push image with the same marker. Deploy → pod runs with the marker. Assert Cortex links all three.

```
git tag(commit, CortexSim-{nonce}) → build(image with LABEL=CortexSim-{nonce}) → deploy(pod with annotation) → query(traceability) → assert(chain.length = 3)
```

**Best fit TCs.** TC-ERV-05, TC-ASPM-01 (the only TC in F9 directly, but pattern reused by ASPM compliance TCs). The deep-dive is folded into [`methodology-multi-plane-stitching.md`](methodology-multi-plane-stitching.md) §6.

### F10 — Qualitative Evidence

**Pattern.** No automated measurement. DC operator runs a demo workflow, captures a screenshot or screen recording, marks `Outcome Achieved = true` in the run record. The methodology is procedural — a runbook in `pov-scripts/` plus an `attestation` field on Result.

**Best fit TCs.** TC-MDR-01 service-delivery TCs, TC-MTH-01 hunt-report cadence. Useful for managed-service POVs where the deliverable IS the analyst experience, not a timing number.

## 5 · Required schema extensions

The current `scenarios/_schema.yml` covers structural fields but does not encode the v2.0 KPI vocabulary. To make TCs achievable end-to-end (run → KPI → POV report), the schema needs these additions:

```yaml
# ── v2.0 KPI / Methodology fields (new) ─────────────────────────────────────

validation_methodology: "Causality Graph Stitching"
# (optional) Free-text label matching the v2.0 master sheet's column.
# Used in POV reports + UI to surface "what this scenario is testing".

methodology_family: "F2"
# (optional) Family ID from the master methodology doc (F1..F10).
# Drives which validation harness the orchestrator invokes.

primary_kpi: "Cross-Source Correlation Rate"
# (optional) The Primary KPI column from the master sheet.

threshold:
  kpi: "Cross-Source Correlation Rate"
  op: "≥"
  value: 80
  unit: "%"
# (optional) Structured threshold. Replaces the v1 string format.

success_criteria: |
  Multi-source signals (NDR session + EDR process + ITDR auth) stitch into a
  single incident_id within 60s, with causality chain covering all 3 planes.
# (optional) Multi-line natural-language success criteria from the sheet.

moat_tier: "MOAT"
# (optional) One of: MOAT | LEAD | PARITY. Used by POV report generator to
# prioritize differentiated TCs in the executive summary.

# ── Per-expected-detection additions ───────────────────────────────────────

steps:
  - id: step-01
    expected_detections:
      - plane: NDR
        type: Analytics
        description: "..."
        verification_xql: |
          dataset = pan_ngfw_traffic | filter ...
        # (new optional) The XQL query that proves this detection fired.
        # Validation harness runs this on a timer and auto-marks observed.
        kpi_contribution:
          kpi: MTTD
          op: "≤"
          value: 60
          unit: seconds
        # (new optional) How this detection contributes to the scenario KPI.
```

These fields are **all optional** so existing scenarios continue to load. The Pydantic loader treats them as enrichment; the orchestrator uses them to drive the right harness if present, falls back to the existing manual-validate flow if not.

The schema change + Pydantic update is scoped as a separate task (`Extend scenario schema for validation methodology fields`) — see §6.

## 7 · How to use this document during a POV

For a Domain Consultant scoping a POV:

1. **Customer names the use cases they want validated.** Map to UC IDs in [`v2.0-tc-mapping-table.csv`](v2.0-tc-mapping-table.csv).
2. **Filter to MOAT-tier TCs first** (79 of 248) — those are the ones where Cortex meaningfully outpositions CrowdStrike/MSFT/SentinelOne, and the master sheet documents the positioning verbatim in the `CrowdStrike Position` / `Microsoft Position` / `SentinelOne Position` columns.
3. **Look up the methodology family** for each selected TC (column `Family` in the CSV). The family tells you which CortexSim harness will validate it.
4. **Check the scenario file** (column `Scenario dir`). If it exists → reusable. If marked `(proposed)` → this TC needs a scenario authored before the POV can run it.
5. **For each scenario, capture `primary_kpi` + `threshold` in the Run record.** The POV report exporter (`GET /api/runs/{id}/report?format=markdown`) will then emit a Cortex-branded report with the KPI evidence inline, ready for the customer-facing readout.

The end-state is a POV where every TC the customer asked for has: a scenario YAML that runs it, a Run record with timestamped Results, a KPI number with a pass/fail verdict, and a competitive-positioning paragraph already written into the v2.0 master sheet's positioning columns.
